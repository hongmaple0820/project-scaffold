# 团队协作规范

**版本**: v1.0  
**适用范围**: 所有项目团队

---

## 1. 角色与职责

| 角色 | 主要职责 |
|------|----------|
| **技术负责人** | 技术决策、代码审查、架构把控 |
| **架构师** | 系统设计、技术选型、规范制定 |
| **开发工程师** | 功能开发、单元测试、Bug修复 |
| **测试工程师** | 测试用例、自动化测试、质量把控 |
| **产品经理** | 需求分析、优先级排序、验收 |

---

## 2. 沟通渠道

| 渠道 | 用途 | 响应时间 |
|------|------|----------|
| **即时通讯** | 紧急问题、快速确认 | 15分钟 |
| **邮件** | 正式通知、决策记录 | 24小时 |
| **项目管理工具** | 任务跟踪、需求管理 | 按排期 |
| **文档协作** | 设计文档、知识沉淀 | 异步 |
| **PR评论** | 代码级讨论 | 4小时 |
| **站会** | 进度同步 | 当面 |

---

## 3. 会议规范

| 会议 | 频率 | 时长 | 产出 |
|------|------|------|------|
| **站会** | 每日 | 15min | 同步进度 |
| **迭代计划** | 每迭代 | 2h | 迭代计划 |
| **技术评审** | 按需 | 1h | 技术方案 |
| **代码审查会** | 按需 | 1h | 质量提升 |
| **回顾会** | 每迭代 | 1h | 改进措施 |

**会议纪律**:
- 提前5分钟到场
- 明确议程和目标
- 指定记录人
- 24小时内发纪要

---

## 4. 任务协作

### 4.1 粒度

| 类型 | 粒度 | 耗时 |
|------|------|------|
| **Epic** | 大功能模块 | 1-2月 |
| **Story** | 可交付功能 | 1-2周 |
| **Task** | 可执行单元 | 1-3天 |
| **Sub-task** | 技术步骤 | 2-4h |

### 4.2 状态流转

```
待开发 → 开发中 → 待测试 → 测试中 → 待发布 → 已完成
```

### 4.3 卡片规范

```
[类型] 描述

例:
[feat] 添加用户登录
[fix] 修复文件上传超时
[docs] 更新API文档
```

---

## 5. 代码协作

### 5.1 代码所有权

- 主要负责人对代码质量负责
- 修改他人模块需告知负责人
- 紧急情况可绕过，事后同步

### 5.2 审查分配

- 日常PR: 1人审查
- 核心模块PR: 2人审查
- 架构变更PR: 架构组评审

### 5.3 冲突解决

**技术分歧**:
1. 小范围讨论（2-3人）
2. 无法达成一致升级到技术负责人
3. 重大决策走架构评审
4. 记录到 ADR

---

## 6. 知识共享

### 6.1 必须沉淀

- 技术方案评审文档
- 疑难问题解决过程
- 性能优化经验
- 故障复盘报告

### 6.2 技术分享

| 形式 | 频率 | 内容 |
|------|------|------|
| 技术分享会 | 双周 | 新技术、最佳实践 |
| 代码走读 | 每月 | 核心代码讲解 |
| 故障复盘 | 按需 | 线上问题分析 |

### 6.3 新人培养

**第一周**:
- 开发环境搭建
- 阅读核心文档
- 导师分配

**第一个月**:
- 完成3个简单任务
- 参与代码审查
- 技术分享旁听
