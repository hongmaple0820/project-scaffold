# 通用团队协作规范

**版本**: v1.0  
**适用范围**: 所有项目团队

---

## 1. 团队角色与职责

### 1.1 角色定义

| 角色 | 主要职责 | 关注点 |
|------|----------|--------|
| **技术负责人** | 技术决策、代码审查、架构把控 | 技术质量、团队成长 |
| **架构师** | 系统设计、技术选型、规范制定 | 可扩展性、可维护性 |
| **开发工程师** | 功能开发、单元测试、Bug修复 | 功能实现、代码质量 |
| **测试工程师** | 测试用例、自动化测试、质量把控 | 功能正确性、稳定性 |
| **产品经理** | 需求分析、优先级排序、验收 | 业务价值、用户体验 |
| **项目经理** | 进度跟踪、资源协调、风险管控 | 按时交付、团队效率 |

### 1.2 职责矩阵 (RACI)

| 活动 | 技术负责人 | 架构师 | 开发 | 测试 | 产品经理 |
|------|-----------|--------|------|------|----------|
| 需求评审 | C | C | R | C | A |
| 技术方案 | A | R | C | I | I |
| 代码开发 | I | C | R | I | I |
| 代码审查 | A | C | R | I | I |
| 测试执行 | I | I | C | R | C |
| 发布上线 | A | C | R | R | I |

**R-负责执行, A-最终负责, C-咨询, I-知会**

## 2. 沟通协作规范

### 2.1 沟通渠道

| 渠道 | 用途 | 响应时间 | 示例 |
|------|------|----------|------|
| **即时通讯** (钉钉/飞书) | 紧急问题、快速确认 | 15分钟 | "线上问题紧急处理" |
| **邮件** | 正式通知、决策记录 | 24小时 | "架构评审会议纪要" |
| **项目管理工具** (Jira/Linear) | 任务跟踪、需求管理 | 按排期 | 用户故事、Bug单 |
| **文档协作** (飞书/Confluence) | 设计文档、知识沉淀 | 异步 | 技术方案、规范 |
| **代码审查** (PR评论) | 代码级讨论 | 4小时 | 具体代码行评论 |
| **站会** | 进度同步、阻塞暴露 | 当面 | 每日15分钟站会 |

### 2.2 会议规范

**会议类型**:

| 会议 | 频率 | 时长 | 参与者 | 产出 |
|------|------|------|--------|------|
| **站会** | 每日 | 15min | 全员 | 同步进度、暴露阻塞 |
| **迭代计划会** | 每迭代 | 2h | 全员 | 迭代计划、任务分解 |
| **技术评审会** | 按需 | 1h | 相关开发+架构 | 技术方案确认 |
| **代码审查会** | 按需 | 1h | 核心成员 | 代码质量提升 |
| **迭代回顾会** | 每迭代 | 1h | 全员 | 改进措施 |
| **架构评审会** | 按月 | 2h | 架构组 | 架构决策 |

**会议纪律**:
- 提前5分钟到场
- 明确议程和目标
- 指定记录人
- 超时30分钟自动散会
- 24小时内发出会议纪要

### 2.3 异步协作

**原则**: 能用异步解决的不用开会

**异步协作方式**:
1. **文档评论** - 在文档中标注问题，@相关人员
2. **PR描述** - 详细描述变更，@审查者
3. **Issue讨论** - 技术问题在Issue中讨论，留痕
4. **录屏分享** - 复杂演示录屏分享，不占用大家时间

## 3. 任务协作规范

### 3.1 任务粒度

| 类型 | 粒度 | 耗时 | 示例 |
|------|------|------|------|
| **史诗 (Epic)** | 大功能模块 | 1-2月 | "网盘聚合服务" |
| **故事 (Story)** | 可交付功能 | 1-2周 | "百度网盘接入" |
| **任务 (Task)** | 可执行单元 | 1-3天 | "实现OAuth授权" |
| **子任务** | 技术步骤 | 2-4h | "编写授权URL接口" |

### 3.2 任务状态流转

```
待开发 (Backlog)
    ↓ 开发认领
开发中 (In Progress)
    ↓ 开发完成
待测试 (Ready for Test)
    ↓ 测试通过
测试中 (In Test)
    ↓ 验收通过
待发布 (Ready for Release)
    ↓ 发布上线
已完成 (Done)
```

### 3.3 任务卡片规范

**标题格式**:
```
[<类型>] <简明描述>

例:
[feat] 百度网盘OAuth授权
[fix] 修复文件上传超时
[docs] 更新API使用文档
```

**描述模板**:
```markdown
## 背景
说明为什么要做这个任务

## 目标
明确任务完成标准

## 验收标准
- [ ] 标准1
- [ ] 标准2

## 相关文档
- [设计文档](link)
- [API定义](link)

## 预估工时
3天
```

## 4. 代码协作规范

### 4.1 代码所有权

| 模块 | 主要负责人 | 备份人员 |
|------|-----------|----------|
| 认证模块 | @user1 | @user2 |
| 文件模块 | @user2 | @user3 |
| 驱动层 | @user3 | @user1 |

**规则**:
- 主要负责人对代码质量负责
- 修改他人模块需告知负责人
- 紧急情况下可绕过，但需事后同步

### 4.2 代码审查分配

**审查人选择**:
- 日常PR: 1人审查（模块负责人或团队成员）
- 核心模块PR: 2人审查（技术负责人+模块负责人）
- 架构变更PR: 架构组评审

**审查时间要求**:
- 普通PR: 4小时内响应
- 紧急PR: 1小时内响应
- 超时未响应: 提醒或换人

### 4.3 冲突解决

**代码冲突**:
```
1. 自行解决简单冲突
2. 复杂冲突寻求原作者协助
3. 冲突频繁时考虑架构调整
```

**技术分歧**:
```
1. 小范围讨论（2-3人）
2. 无法达成一致升级到技术负责人
3. 重大决策走架构评审流程
4. 决策后记录到 ADR (Architecture Decision Record)
```

## 5. 知识共享规范

### 5.1 知识沉淀

**必须沉淀**:
- 技术方案评审文档
- 疑难问题解决过程
- 性能优化经验
- 故障复盘报告

**沉淀位置**:
```
docs/skills/           # 技术技能
docs/reference/adr/    # 架构决策
docs/reference/issues/ # 问题复盘
```

### 5.2 技术分享

**分享形式**:
| 形式 | 频率 | 时长 | 内容 |
|------|------|------|------|
| 技术分享会 | 双周 | 30-60min | 新技术、最佳实践 |
| 代码走读 | 每月 | 60min | 核心代码讲解 |
| 故障复盘 | 按需 | 30min | 线上问题分析 |
| 读书分享 | 季度 | 60min | 技术书籍/文章 |

**分享流程**:
1. 提前3天通知主题
2. 准备PPT或文档
3. 录制回放（可选）
4. 归档到知识库

### 5.3 新人培养

**入职第一周**:
- [ ] 开发环境搭建
- [ ] 代码库访问权限
- [ ] 阅读核心文档
- [ ] 导师分配

**第一个月**:
- [ ] 完成3个简单任务
- [ ] 参与代码审查
- [ ] 技术分享旁听
- [ ] 转正答辩准备

**导师职责**:
- 每周至少1次1on1
- 代码审查优先
- 解答技术和业务问题
- 评估成长情况

## 6. 质量协作规范

### 6.1 质量门禁

**开发阶段**:
- [ ] 单元测试覆盖率>80%
- [ ] 代码审查通过
- [ ] 静态检查无高优问题

**测试阶段**:
- [ ] 功能测试通过
- [ ] 回归测试通过
- [ ] 性能测试达标

**发布阶段**:
- [ ] 发布 checklist 完成
- [ ] 监控配置完成
- [ ] 回滚方案就绪

### 6.2 Bug管理

**严重程度**:
| 级别 | 定义 | 响应时间 | 修复时间 |
|------|------|----------|----------|
| P0-致命 | 系统不可用 | 立即 | 4小时 |
| P1-严重 | 核心功能故障 | 1小时 | 24小时 |
| P2-一般 | 非核心功能故障 | 4小时 | 本周内 |
| P3-轻微 | 体验问题 | 24小时 | 下迭代 |

**Bug流程**:
```
发现 → 记录 → 确认 → 分配 → 修复 → 验证 → 关闭
```

### 6.3 线上问题协作

**响应流程**:
```
1. 发现者立即在群里通知（@所有人）
2. 技术负责人组织响应
3. 定位问题、评估影响
4. 修复或回滚
5. 事后复盘、记录
```

**值班制度**:
- 每日值班人员负责监控
- 值班表提前一周公布
- 节假日顺延或指定替班

## 7. 跨团队协作

### 7.1 依赖管理

**识别依赖**:
- 在迭代计划会识别跨团队依赖
- 明确依赖方和被依赖方
- 约定交付时间和接口

**依赖跟踪**:
- 在项目管理工具中标注依赖
- 每周检查依赖状态
- 阻塞时升级协调

### 7.2 接口协作

**接口定义**:
- 前后端共同评审API设计
- 先定义接口再开发
- 接口变更需通知相关方

**Mock服务**:
- 后端延迟时前端可用Mock
- Mock数据由后端提供
- 完成后替换为真实接口

### 7.3 冲突协调

**优先级冲突**:
- 双方技术负责人协商
- 无法解决升级到更高层
- 以业务价值为最终标准

**资源冲突**:
- 提前规划资源需求
- 错峰使用共享资源
- 必要时申请加人

## 8. 工具与平台

### 8.1 推荐工具链

| 用途 | 工具 | 说明 |
|------|------|------|
| 代码托管 | GitLab/GitHub | 私有仓库 |
| 项目管理 | Jira/Linear | 敏捷管理 |
| 文档协作 | 飞书/Confluence | 知识库 |
| 即时通讯 | 钉钉/飞书 | 日常沟通 |
| 会议 | Zoom/腾讯会议 | 远程会议 |
| CI/CD | Jenkins/GitHub Actions | 自动化 |
| 监控 | Grafana/Prometheus | 系统监控 |
| 日志 | ELK/Loki | 日志分析 |

### 8.2 账号权限管理

**账号规范**:
- 使用公司邮箱注册
- 实名显示
- 离职及时清理权限

**权限分级**:
- Owner: 最高权限，可删除仓库
- Maintainer: 可管理仓库设置
- Developer: 可推送代码
- Reporter: 可查看代码和Issue

## 9. 工作纪律

### 9.1 工作时间

**核心时间**: 10:00 - 18:00  
**弹性时间**: 允许±2小时弹性

**响应要求**:
- 工作时间内及时响应
- 紧急问题随时响应
- 请假提前告知并交接

### 9.2 远程工作

**适用场景**:
- 每周2天可申请远程
- 特殊情况可临时申请
- 新人首月建议 onsite

**远程要求**:
- 保持在线状态
- 会议必须视频
- 及时回复消息

### 9.3 请假与交接

**请假流程**:
1. 提前在系统提交申请
2. 通知直接上级
3. 群里告知团队
4. 完成工作交接

**交接内容**:
- 进行中的任务状态
- 阻塞的问题
- 需要关注的事项
- 紧急联系人
