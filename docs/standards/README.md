# 技术规范体系说明

**版本**: v1.0  
**日期**: 2026-05-22

---

## 规范分层架构

```text
docs/standards/
├── common/                    # 通用规范（所有项目复用）
│   ├── NAMING.md             # 通用命名规范
│   ├── API_STANDARDS.md      # 通用API规范
│   ├── DATABASE_STANDARDS.md # 通用数据库规范
│   ├── GIT_STANDARDS.md      # 通用Git规范
│   ├── DEPLOYMENT_STANDARDS.md # 部署与环境配置规范
│   └── CODE_REVIEW.md        # 通用代码审查规范
│
├── projects/                  # 项目特定规范
│   └── <service-name>/        # 具体微服务（基于脚手架创建时填充）
│       ├── PROJECT_SPEC.md   # 项目专属规范
│       └── ARCHITECTURE.md   # 项目架构文档
│
└── README.md                  # 规范入口（本文件）
```

---

## 如何使用这套规范

### 场景1：新启动一个微服务

1. **继承通用规范**：直接遵循本脚手架 `common/` 目录下的所有通用规范。
2. **创建项目特定规范**：在 `projects/<微服务名>/` 下创建专属规范。
3. **记录差异**：项目级规范引用通用规范，并且仅记录差异点或补充细节。

**示例 - 新服务规范模板**:

```markdown
# XXX微服务规范

## 1. 规范继承

本项目遵循脚手架的以下通用规范：
- [通用命名规范](../common/NAMING.md)
- [通用API规范](../common/API_STANDARDS.md)
- [通用数据库规范](../common/DATABASE_STANDARDS.md)

## 2. 项目特定规范

### 2.1 技术栈
（与通用规范不同的部分，如引入了特定的图数据库）

### 2.2 目录结构
（微服务特有的模块结构）

### 2.3 命名约定
（与通用规范的差异）
```

### 场景2：现有项目接入规范

1. **评估现状**：对照通用规范，列出差异
2. **渐进式整改**：按优先级逐步对齐
3. **记录例外**：在项目中记录不遵循通用规范的原因

### 场景3：跨微服务协作

- 所有微服务共享并遵守 `common/` 下的通用规范
- 接口对接与协议交互时，必须以通用规范为唯一基准

## 规范级别定义

| 级别 | 说明 | 示例 |
|------|------|------|
| **强制(MUST)** | 所有基于本脚手架的微服务必须遵守 | 数据库字段命名、统一响应格式 |
| **推荐(SHOULD)** | 默认遵守，可说明原因后偏离 | 目录结构、错误码范围 |
| **可选(MAY)** | 根据项目特点选择 | 具体技术栈选型细分、缓存策略 |

---

## 规范维护流程

1. **通用规范变更**：需向架构组提交脚手架 PR 评审，影响所有基于此脚手架的项目。
2. **项目特定规范变更**：各微服务负责人审批，只影响当前微服务。
3. **例外申请**：向架构组提交例外申请，记录在项目文档中。
